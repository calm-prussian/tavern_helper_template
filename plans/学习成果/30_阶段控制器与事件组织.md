# 阶段控制器与事件组织

## 1. 文档目标

本文用于说明角色卡中的“多阶段控制器”“阶段条目”“即时事件”“剧情分支事件”应如何组织，并沉淀为可独立使用的长期模式。

本文不依赖一次性样本目录，而是把学到的结构模式提炼为通用方法，便于后续直接复用。

## 2. 什么是阶段控制器

阶段控制器的核心职责是：

- 根据当前变量状态
- 自动决定应启用哪一组阶段内容
- 把对应阶段条目注入当前上下文

也就是说，阶段控制器本身不负责写完整人物设定，而是负责“选择哪段设定应该生效”。

推荐把阶段控制器理解为：

- 世界书里的动态分发器
- 变量到人设条目的桥梁
- 静态阶段条目的调度入口

## 3. 什么时候适合使用阶段控制器

以下情况推荐使用：

- 角色关系会分阶段变化
- 世界状态会显著改变角色表达方式
- 有明确数值门槛，如好感度、堕落值、依存度、怀疑度
- 同一角色在不同阶段说话风格差异大
- 希望避免把所有阶段信息同时塞给模型

以下情况可以不使用：

- 角色状态变化很小
- 只需要少量动态字段即可表达差异
- 世界书规模很小，不值得维护额外控制器

## 4. 阶段系统的推荐组成

一个完整的阶段系统通常由四部分组成：

1. 阶段相关变量
2. 控制器条目
3. 阶段条目
4. 与阶段相关的事件或补充规则

推荐结构如下：

```text
世界书/
├── 角色/
│   ├── 角色A_多阶段控制器.txt
│   ├── 角色A_阶段01_xxx.txt
│   ├── 角色A_阶段02_xxx.txt
│   ├── 角色A_阶段03_xxx.txt
│   └── ...
├── 事件/
│   ├── 事件A.yaml
│   └── 事件B.yaml
└── 变量/
    ├── initvar.yaml
    └── 变量更新规则.yaml
```

## 5. 阶段变量的推荐设计

### 5.1 阶段变量最好来自已有核心数值

推荐优先让阶段控制器依赖这些已经存在的核心数值：

- 好感度
- 信赖值
- 依存度
- 堕落值
- 世界分支
- 任务完成状态
- 关键布尔条件

如果已有变量足够表达阶段，不一定需要额外新增一个显式“阶段名”字段。

### 5.2 何时增加显式阶段字段

如果以下情况成立，推荐增加显式阶段字段：

- 需要在界面中显示当前阶段
- 阶段不只由单个数值决定
- 阶段切换逻辑复杂且会被多处使用
- 某些事件只在特定阶段生效

否则，直接在控制器内部按条件判断即可。

## 6. 阶段控制器的推荐写法

控制器的基本模式是：

1. 读取变量
2. 根据条件分支
3. 注入对应阶段条目

推荐结构大致如下：

```ejs
<%_
const value = getvar('stat_data.角色.某数值', { defaults: 0 });
_%>

<%_ if (value >= 80) { _%>
<%- await getwi(null, '角色_阶段04_终局期') %>
<%_ } else if (value >= 50) { _%>
<%- await getwi(null, '角色_阶段03_变化期') %>
<%_ } else { _%>
<%- await getwi(null, '角色_阶段01_初始期') %>
<%_ } _%>
```

### 6.1 控制器职责应保持单一

推荐控制器只做“选择条目”这件事，不要在控制器中塞入过多叙事正文。

原因是：

- 控制器应易读
- 条件逻辑应清晰
- 阶段内容应独立维护

### 6.2 阈值应与变量规则对应

如果控制器使用的是：

- 好感度 0~200
- 堕落值 -100~100
- 依存度 0~100

那么阶段阈值应与 `变量更新规则` 的幅度和变化逻辑相匹配，否则模型体验会不稳定。

## 7. 阶段条目的推荐写法

阶段条目的职责是描述：

- 当前阶段下角色的说话方式
- 当前阶段下角色的态度边界
- 当前阶段下角色更容易采取的行为
- 当前阶段下角色的互动禁区与开放区

推荐每个阶段条目都聚焦“当前阶段的差异”，而不是重复写全部基础设定。

### 7.1 阶段条目不应重复基础资料

基础资料应放在：

- 角色基础条目
- 世界观条目
- 角色详情条目

阶段条目只补充“当前阶段相对前一阶段有哪些改变”。

### 7.2 阶段条目应可叠加理解

推荐让模型先获得角色基础，再通过控制器额外获得当前阶段内容。

这样做的优点是：

- 阶段条目更短
- 不容易写崩角色底层一致性
- 后续改某个阶段时不会牵连全部资料

## 8. 单维阶段与多维阶段

### 8.1 单维阶段

如果阶段只由一个核心数值决定，例如：

- 好感度
- 信赖值
- 依存度

那么推荐使用单维阶段控制器。优点是简单、稳定、直观。

### 8.2 多维阶段

如果角色状态同时受多个变量影响，例如：

- 好感度 + 堕落值
- 信赖值 + 任务分支
- 世界分支 + 婚姻状态 + 关键认知

则推荐使用多维判断控制器。

此时建议：

- 先判断优先级最高的分支
- 再判断次级阶段
- 避免出现大量重叠条件相互打架

## 9. 世界分支型控制器

除了角色阶段外，控制器也可以用于剧情分支调度。

典型适用场景：

- 已知关键事件是否发生
- 角色是否接受玩家
- 是否结婚
- 是否到达某个时间节点
- 是否进入某条特殊支线

推荐把这种控制器理解为“剧情分支路由器”。

### 9.1 推荐使用单独的关键条件组

如果剧情分支复杂，推荐在变量中显式保留一组关键条件，例如：

- 是否知晓真相
- 是否进入婚姻
- 是否完成任务
- 是否到达关键日期

这样控制器更易维护，也方便状态栏展示。

## 10. 事件组织的推荐方法

事件通常有两种：

### 10.1 即时事件

即时事件适用于：

- 某个条件一满足就应立即发生提示或效果
- 某个提示词只在特殊状态下注入
- 某个极端变量值需要插入补充引导

推荐把即时事件做成单独世界书条目或脚本注入逻辑。

### 10.2 历史/剧情事件

历史事件或剧情事件适用于：

- 记录发生过的重要事情
- 作为未来剧情回溯依据
- 影响角色态度与世界状态
- 推动后续阶段切换

推荐把这类事件与变量状态联动，而不是只写在世界书正文里让模型自己记。

## 11. 事件与变量的配合方式

推荐形成以下闭环：

1. 事件发生
2. 变量更新
3. 控制器读取变量
4. 阶段条目切换
5. 状态栏或界面同步更新

如果少了变量这一层，事件与阶段切换就容易依赖模型短期记忆，不稳定。

## 12. 事件命名与目录组织建议

推荐使用可读性强的命名方式，例如：

- `事件_雨夜摊牌`
- `事件_第一次共同作战`
- `事件_角色A得知真相`

如果事件很多，推荐再按职责分组：

- `世界书/事件/主线/`
- `世界书/事件/支线/`
- `世界书/事件/即时/`
- `世界书/事件/历史锚点/`

## 13. 阶段系统与状态栏的关系

如果角色卡有状态栏，推荐让状态栏读取以下任意一种：

- 核心数值本身
- 派生阶段字段
- 控制器对应的阶段描述

推荐不要让状态栏自己重复实现一套阶段判断逻辑，除非只是简单视觉辅助。

更理想的方式是：

- 阶段逻辑主要在变量或控制器层统一定义
- 状态栏只做读取与展示

## 14. 即时事件脚本的推荐职责

如果用脚本处理即时事件，推荐脚本职责聚焦在：

- 根据变量条件决定是否注入提示
- 根据阈值切换附加提示词
- 避免把核心角色设定写进脚本

也就是说，脚本更适合做“触发器”，不适合替代世界书主体。

## 15. 推荐的构建顺序

如果角色卡需要多阶段系统，推荐按以下顺序构建：

1. 先定义关键变量
2. 再定义阶段划分标准
3. 再写阶段条目
4. 再写控制器条目
5. 最后补即时事件或特殊事件脚本

这样做的优点是：

- 阶段边界更清晰
- 条目不会先写后推翻
- 控制器条件更容易对齐变量规则

## 16. 常见误区

### 16.1 阶段条目重复写完整角色卡

这会导致维护成本极高。推荐基础资料和阶段差异分开。

### 16.2 控制器内嵌大段设定正文

控制器应负责调度，不应成为第二份角色说明书。

### 16.3 没有变量支撑却硬做阶段系统

如果没有稳定变量来源，阶段切换很容易漂移。推荐先补变量模型。

### 16.4 事件很多但没有分组

事件一多，如果没有组织层次，会导致后期维护困难。推荐尽早分类。

### 16.5 阶段阈值与变量变化幅度不匹配

如果变量每次变化幅度极小，但阶段阈值极大，阶段切换会过慢；反之会过快。推荐统一规划。

## 17. 一句话总结

阶段控制器与事件组织的推荐方式是：**用变量保存真实状态，用控制器按状态选择阶段，用阶段条目提供当前行为边界与风格差异，再让事件推动变量变化，从而形成“事件→变量→控制器→阶段→显示”的稳定闭环。**
