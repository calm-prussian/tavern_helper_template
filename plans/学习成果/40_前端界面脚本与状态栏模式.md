# 前端界面、脚本与状态栏模式

## 1. 文档目标

本文用于沉淀本项目中与前端界面、脚本、流式楼层界面、状态栏相关的推荐模式，确保后续即使删除旧学习素材，也仍然能独立判断：

- 什么时候该写脚本
- 什么时候该写前端界面
- 什么时候该写流式楼层界面
- 什么时候适合用正则状态栏
- 什么时候应该升级为 Vue 状态栏

## 2. 三类前端能力的基本定位

### 2.1 脚本

脚本是后台增强能力，适合：

- 监听事件
- 调整消息楼层
- 读写变量
- 注入提示词
- 挂载设置界面
- 用 jQuery 操作酒馆页面

如果一个功能主要是“逻辑驱动”而不是“界面驱动”，推荐优先用脚本。

### 2.2 前端界面

前端界面是显示在消息楼层中的独立界面，适合：

- 展示状态栏
- 展示富交互面板
- 展示卡片型内容
- 在楼层中提供按钮或选择器
- 对某条消息做更美观的结构化展示

如果一个功能主要是“显示与交互”，推荐优先用前端界面。

### 2.3 流式楼层界面

流式楼层界面是特殊脚本方案，适合：

- 需要在模型流式输出时同步渲染界面
- 需要替换默认楼层显示方式
- 需要边输出边解析结构内容
- 需要对流式文本进行高亮、分段或局部交互

如果必须在输出未结束前就渲染内容，推荐考虑流式楼层界面。

## 3. 如何判断该选哪一种

### 3.1 优先选脚本的情况

推荐在以下场景优先选脚本：

- 只需要监听酒馆事件
- 只需要改变量或楼层内容
- 只需要提供设置面板
- 只需要注入提示词或按钮
- 不需要独立楼层 UI

### 3.2 优先选前端界面的情况

推荐在以下场景优先选前端界面：

- 需要一个完整的状态栏或消息内 UI
- 需要多个组件协作展示数据
- 需要比纯文本更清晰的排版
- 需要用户点击、切换、分页、筛选

### 3.3 优先选流式楼层界面的情况

推荐在以下场景优先选流式楼层界面：

- 需要在模型尚未输出完时就实时展示界面
- 需要边输出边做结构化处理
- 需要把流式内容替换成自定义视觉呈现

如果只是普通状态栏或普通楼层增强，不推荐一开始就上流式方案。

## 4. 脚本的推荐结构

最小脚本项目通常只有一个 [`index.ts`](初始模板/脚本/新建为src文件夹中的文件夹/index.ts)。

推荐组织方式：

- `index.ts` 作为入口
- 按功能拆成多个小文件
- 在入口统一导入这些功能文件

例如推荐模式：

- 加载/卸载逻辑
- 按钮注册
- 设置界面
- 事件监听
- 消息调整逻辑

这种模式可参考 [`示例/脚本示例/index.ts`](示例/脚本示例/index.ts:1)。

## 5. 脚本的生命周期推荐写法

### 5.1 加载时机

推荐始终使用 jQuery 的：

- `$(() => { ... })`

而不是使用：

- `DOMContentLoaded`
- 顶层直接执行复杂逻辑

因为本项目里的脚本很多时候是通过远程 `import` 动态载入的，`DOMContentLoaded` 并不可靠。

### 5.2 卸载时机

推荐使用：

- `$(window).on('pagehide', ...)`

而不是：

- `unload`

这样更符合项目当前的加载/卸载方式。

## 6. 脚本设置界面的推荐模式

如果脚本需要用户设置，推荐：

1. 用 zod 定义设置结构
2. 用 Pinia 封装设置状态
3. 从脚本变量读取和持久化
4. 用 Vue 组件提供设置界面

推荐职责分工如下：

- `settings.ts`：负责状态与持久化
- `设置界面.vue`：负责展示与交互
- `设置界面.ts`：负责把设置界面挂到酒馆设置区域

### 6.1 设置数据的推荐存储方式

推荐把脚本设置存到：

- `type: 'script'`
- `script_id: getScriptId()`

### 6.2 推荐的状态同步方式

推荐：

- 先用 `Schema.parse(...)` 读取数据
- 再用 `watchEffect(...)` 进行回写
- 回写前用 `klona(...)` 去掉响应式代理

这样状态结构更稳定，也更适合长期维护。

## 7. 前端界面的推荐结构

普通前端界面推荐至少包含：

- [`index.html`](初始模板/前端界面/新建为src文件夹中的文件夹/index.html)
- [`index.ts`](初始模板/前端界面/新建为src文件夹中的文件夹/index.ts)
- [`App.vue`](初始模板/前端界面/新建为src文件夹中的文件夹/App.vue)

其中：

### 7.1 [`index.html`](初始模板/前端界面/新建为src文件夹中的文件夹/index.html)

应保持极简，通常只保留：

- 空的 `<head>`
- 一个 `#app` 容器

推荐不要在这里写本地脚本引用或本地样式引用。

### 7.2 [`index.ts`](初始模板/前端界面/新建为src文件夹中的文件夹/index.ts:1)

负责：

- 创建 Vue 应用
- 挂载 `Pinia`
- 挂载根组件
- 注册卸载逻辑

### 7.3 [`App.vue`](初始模板/前端界面/新建为src文件夹中的文件夹/App.vue)

负责：

- 承载主要模板
- 组织组件结构
- 处理展示逻辑
- 做局部样式

## 8. 状态栏的两种主路线

### 8.1 纯正则状态栏

这是较旧但仍可用的方案。

适合场景：

- 需求简单
- 只是展示少量信息
- 临时验证样式
- 不想建立完整前端工程

优点：

- 上手快
- 文件少
- 对简单卡足够用

缺点：

- 维护困难
- 样式与逻辑耦合重
- 很难扩展成复杂交互
- 组件复用性差

### 8.2 Vue 状态栏

这是更推荐的长期方案。

适合场景：

- 状态栏内容较多
- 有多个分区或页签
- 需要响应式更新
- 需要多个组件协作
- 后期预计持续维护

优点：

- 结构清晰
- 易于拆组件
- 易于复用样式和逻辑
- 更适合与 MVU 数据联动

缺点：

- 初始搭建成本略高
- 需要更多文件组织

## 9. 什么时候该从正则状态栏升级到 Vue 状态栏

如果出现以下任一情况，推荐考虑升级：

- 状态栏开始有多个角色面板
- 状态栏开始有页签切换
- 状态栏需要复杂样式与动画
- 状态栏逻辑已明显超过简单字符串填充
- 状态栏要复用变量读取逻辑

如果只是单块静态展示，没有必要强行升级。

## 10. Vue 状态栏的推荐结构

推荐结构如下：

```text
界面/状态栏/
├── index.html
├── index.ts
├── App.vue
├── store.ts
└── components/
    ├── PanelA.vue
    ├── PanelB.vue
    └── ...
```

### 10.1 [`store.ts`](示例/角色卡示例/界面/状态栏/store.ts:1)

推荐承担：

- 统一读取 MVU 数据
- 暴露响应式 `store.data`
- 把数据接入整个状态栏组件树

### 10.2 组件层

推荐按界面区域拆组件，而不是按 HTML 标签拆组件。

例如：

- 头部区域
- 世界信息面板
- 角色面板
- 物品栏面板
- 页签导航

## 11. 流式楼层界面的推荐模式

流式楼层界面本质上仍属于脚本项目，只是它会用脚本去替换或接管消息楼层显示。

推荐结构通常包括：

- `index.ts`
- `App.vue`
- 若干局部组件

### 11.1 挂载方式

推荐使用项目现成的流式挂载工具，而不是自己重造挂载链。

### 11.2 数据来源

流式界面推荐通过上下文读取：

- 当前消息文本
- 当前消息 id
- 当前流式阶段信息

### 11.3 适合的处理类型

流式界面尤其适合：

- 结构标签分段显示
- 流式高亮
- 搜索与过滤
- 流式文本的局部替换展示

## 12. 样式设计的推荐原则

### 12.1 普通前端界面

推荐优先：

- Tailwind 类名
- `<style scoped>`

### 12.2 脚本挂载到宿主页面的组件

如果脚本把组件挂到酒馆主页面而不是 iframe 内，推荐谨慎处理样式隔离。

此时应注意：

- 不要随意污染宿主类名
- 需要时使用样式转移工具
- 非必要不要滥用 Tailwind 到宿主页面

### 12.3 状态栏

状态栏通常更适合：

- 宽度自适应
- 不出现横向滚动
- 不用会强撑容器的布局
- 内容块清晰、分区稳定

## 13. 数据读取的推荐做法

### 13.1 纯正则状态栏

推荐直接读变量并填充 DOM。

适合小规模信息显示。

### 13.2 Vue 状态栏

推荐通过统一 `store` 读取数据，再由组件消费。

不推荐在多个组件里散乱直读底层变量接口。

### 13.3 脚本设置界面

推荐通过 `Pinia + zod + script variables` 管理。

## 14. 按需选型的推荐表

### 14.1 只是监听或改写逻辑

推荐：脚本

### 14.2 需要独立消息界面

推荐：前端界面

### 14.3 需要流式渲染

推荐：流式楼层界面

### 14.4 只是简单显示少量状态

推荐：纯正则状态栏或简单前端界面

### 14.5 需要复杂状态栏

推荐：Vue 状态栏

## 15. 常见误区

### 15.1 为简单需求建立过重界面工程

如果只是显示两三条状态，不一定要上完整 Vue 状态栏。

### 15.2 在纯正则状态栏里堆越来越多逻辑

一旦复杂度上来，推荐尽快迁移到前端界面，不要继续在单个 HTML 文本里硬堆。

### 15.3 脚本与界面职责不分

推荐脚本偏逻辑，界面偏展示。两者可以协作，但不要彼此替代到混乱。

### 15.4 流式需求却使用普通前端界面硬撑

普通前端界面不能天然解决流式渲染问题。需要实时显示时，应直接考虑流式方案。

## 16. 一句话总结

前端能力的推荐选型是：**逻辑增强优先脚本，楼层内独立展示优先前端界面，实时流式展示优先流式楼层界面；状态栏简单时可用正则方案，复杂且长期维护时更推荐 Vue 组件化方案。**
