# 风花雪月毕业典礼 — 写卡过程记录

## 项目类型

纯文字角色卡（`index.yaml` + 世界书 + 第一条消息，无 `.ts` 文件）。

## 概念演进

### 初始需求
用户想做《火焰之纹章：风花雪月》题材的毕业典礼 gangbang 角色卡，全员上阵，每人独立条目。

### 第一轮讨论：角色条目范围
- 我提议分组结构（学级文件夹：角色速览/群组/NSFW），参考度假沙滩
- **用户否定**：不要分组，每个人一个独立条目
- 我提议关键词触发（节省上下文）
- **用户否定**：全员蓝灯（缓存原因）

### 第二轮讨论：角色信息怎么写
- 我默认想写外貌+性格+性癖（参考度假沙滩角色条目长度）
- **用户关键提醒**：模型本身就有风花雪月的数据，只强调来源，补充NSFW和性癖即可
- 最终角色条目格式压缩为：来源一行 + 性癖3项 + 群交表现 + 备注

### 第三轮讨论：循环机制
- 用户提到"土拨鼠之日"
- 我误以为是玩法机制（谁记得循环、怎么触发重置）
- **用户澄清**：循环只是为了防AI跑题，把场景固定在这一天
- 最终写为"世界规则"而非"循环约束"，去meta感，明确"黎明→傍晚"重置

### 最终方案确定
- 40个角色条目（黑鹫8+青狮8+金鹿8+教职员6+骑士团3+其他2+灰狼4+特殊1）
- 3个设定条目（世界观、场景、NSFW引导）
- 全部蓝灯常驻
- {{user}}是参与者之一，gangbang平等，不特殊对待

## 设计决策

| 决策 | 选择 | 理由 |

## 踩坑记录

### 1. 角色设定套错原作（严重）
- **金鹿_克劳德**：写成了"沉默送货员型"——套用了FF7克劳德的设定，原作是轻浮开朗的策略家
- **灰狼_尤里**：写成了"俄国少女型"——原作是**男性**（Yuri Leclerc），街头混混气质
- **金鹿_伊古纳兹**：写成了"阴郁魔法师"——原作是温和的艺术爱好者，不是魔法师
- **教训**：即使说"模型有数据"，性癖设定也不能张冠李戴。写之前必须确认角色基本信息（性别、性格、职业）

### 2. 译名混淆
- **教职员_塞洛斯**：写的是Seiros（女神/教团创始人），但用户想要的是Seteth（可操作教职员，绿发严肃男性）
- **教训**：日系游戏译名常有多个版本，写完后必须和用户确认译名，尤其是教职员/神职类角色

### 3. NSFW引导与世界观矛盾
- 初版写"做完不转换话题——嘴里还挂着精液就开始聊**明天的行程**"
- 但世界观设定"时间不会进入第二天"
- **教训**：NSFW引导的每一句话都要和世界观对账，不能有语义冲突

### 4. 场景过度分散导致AI"到处乱跑"
- 初版写了7个地点（宴会厅、花园、大礼堂、宿舍、大教堂、湖边、训练场）
- 第一条消息里写"英谷莉特和希尔凡不见了——湖边传来水声"
- AI会把40个人写得到处分散，破坏gangbang的集中感
- **教训**：gangbang卡必须**锁定单一空间**。用户明确说"一堆人到处乱跑"时，要果断删除多余地点

### 5. 循环表述meta化
- 初版用"循环约束"作为标题，AI理解为剧情机制，会写角色"发现循环""试图打破"
- 改为"世界规则"后，去meta感，AI不再把循环当剧情
- **教训**：防跑题的约束要用"这是背景"而非"这是机制"的表述

### 6. 破折号清理（用户特殊要求）
- 用户要求文本中不要出现破折号（——）
- 40个角色条目+3个设定条目，共43个文件需要批量清理
- **教训**：用户对文本格式可能有特殊偏好，问清楚再写，写完后全局搜索确认

## YAML 格式要点

### 角色条目 YAML（世界书文件）
```yaml
---
角色名:
  来源: 火焰之纹章：风花雪月
  NSFW:
    性癖: ...
    群交表现: ...
    备注: ...
```
- 以 `---` 开头
- 最外层是角色名映射
- 只有4个字段，极简

### index.yaml 条目引用
```yaml
  - 名称: 风花雪月艾黛尔贾特
    启用: true
    激活策略:
      类型: 蓝灯
    插入位置:
      类型: 角色定义之后
      顺序: 101
    激活概率: 100
    递归:
      不可被其他条目激活: true
      不可激活其他条目: true
    文件: 世界书/角色/黑鹫_艾黛尔贾特
```
- `文件:` 不带 `.yaml` 后缀
- 顺序号按学级分组跳跃分配（黑鹫100段、青狮200段、金鹿300段...）
- 分隔条目用向量化类型+禁用，方便酒馆UI中折叠管理

## 关键设计经验

### "模型有数据"型角色卡的核心公式
1. **来源声明要醒目**——在设定条目最前面明确写"本场景所有角色均来源于《XXX》"，让模型立刻调用训练数据
2. **只补模型不知道的部分**——外貌、性格、背景一律不写，只写性癖、群交表现、互动备注
3. **性癖也不能套错**——模型知道角色性格，但不知道你的性癖设定。不能把A角色的性格套到B角色身上（克劳德案）
4. **条目要极简**——每个角色4-5行，40个角色才320行，上下文可控

### 全员蓝灯40+条目的管理技巧
- **分隔条目**：用向量化+禁用状态的条目做学级分界（如"===黑鹫学级开始==="），酒馆UI中可折叠
- **顺序号跳跃分配**：黑鹫100段、青狮200段、金鹿300段... 方便后期插入新角色
- **批量修改**：用 `sed` 批量替换（如清理破折号：`sed -i 's/——/,/g' *.yaml`）

### Gangbang卡的空间锁定
- **必须锁定单一空间**——哪怕世界观里有多个地点，gangbang进行时所有人必须在同一个房间内
- **门要关上**——物理封闭空间，防止AI写"某人出去透气"
- **取消旁观者**——"没有旁观者，每个人都在参与或准备参与"，防止AI写"有人在旁边看"
- **时段推进通过内部细节体现**——蜡烛烧完、光线变暗、衣物堆变高，而不是"换了个房间"

## 给未来 Agent 的提示

1. **先判断项目类型**——纯文字角色卡不需要 MVU/前端/脚本相关内容，跳过
2. **用户说"模型有数据"时**——不要重复写角色外貌性格，只补模型不知道的部分（性癖、群交表现）
3. **蓝灯 vs 绿灯的选择**——问清楚原因，不要擅自建议改触发方式（缓存、上下文等需求用户比你清楚）
4. **循环/重置类约束**——用"世界规则"而非"循环约束"表述，避免meta感；明确写出"A→B"的过渡，不要留空白让AI猜
5. **NSFW引导要与世界观一致**——检查是否有语义矛盾（如"聊明天"vs"不存在明天"）
6. **角色设定必须审核**——即使说"模型有数据"，性癖设定也不能套错角色（克劳德案）
7. **译名要确认**——日系游戏角色译名可能有多个版本（塞洛斯/西提司）
8. **YAML缩进是坑**——`激活策略`和`启用`必须同级，关键字是列表不是逗号分隔字符串
9. **顺序号跳跃分配**——按学级/分组留空段，方便后期插入新角色
10. **分隔条目**——用向量化+禁用状态的条目做分组标记，酒馆UI中可折叠管理
11. **Gangbang卡必须锁定单一空间**——删除多余地点，门要关上，取消旁观者
12. **色色卡开场要快准狠**——20-40行直接建立场景和暧昧基调，不要长篇文艺描写
13. **破折号注意**——用户可能对文本格式有特殊要求，写完后全局搜索确认
14. **avatar.png 是必需的**——bundle前确保头像文件存在且有效，可以从其他项目复制占位
